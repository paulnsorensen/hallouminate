# ADR fm-index-search-signal-001: FM-Index as an additive signal, not an rg replacement  [status: accepted]

- **Context:** Issue #191 flagged lancedb 0.31's FM-Index (PR lancedb/lancedb#3532) as "worth a look" for exact identifier/symbol lookups. Research found hallouminate already patches this exact gap today via an `rg` subprocess (`crates/hallouminate-domain/src/search/ripgrep.rs:1-7`) because LanceDB's BM25 tokenizer misses identifiers with embedded punctuation. Full replacement of `rg` with FM-Index's `contains()` predicate was the initial direction, but Grilling it surfaced two source-verified regressions: `contains()` is documented case-sensitive (`datafusion-functions-53.1.0/src/string/contains.rs:34`) while `rg` runs `--ignore-case`; and FM-Index only sees rows already committed via `merge_insert`, while `rg` scans live disk at query time (losing the "reflects edits before reindex" guarantee).
- **Decision:** Add FM-Index as a fourth, additive ranking signal, entirely internal to `LanceStore::hybrid_search` (`crates/hallouminate-adapters/src/lance.rs`) — a new `contains()`-filtered query, decoded into matched `chunk_id`s, boosting the existing FTS+vector fused output by a flat `CONTAINS_WEIGHT`. `search_with_ripgrep` and its case-insensitive, live-disk `rg` subprocess are unchanged. No `ChunkStore` trait signature change (`hybrid_search`'s `(corpus, query, limit) -> Vec<SearchHit>` shape is preserved, confirmed at `crates/hallouminate-domain/src/indexer/store.rs:15-20`), so no domain-layer or `search.rs` changes are needed.
- **Alternatives:** (a) Replace `rg` outright — rejected per the Grill findings above (real regressions, not hypothetical). (b) Build the FM-Index with no query path yet — rejected as pure speculative infrastructure with no consumer.
- **Consequences:** Two overlapping exact-match mechanisms now exist (`rg` file-level, FM-Index chunk-level) with distinct strengths (case-insensitive + always-fresh vs. faster + format-agnostic + chunk-precise); no existing behavior regresses. `ensure_search_indexes` gains an unconditional FM-Index build on `text` (not row-gated, mirroring FTS).

## Addendum: index-name collision discovered during `/cook`

The coexistence question this ADR left open resolved **negative-then-fixed**, and the actual failure mode was sharper than "might not coexist." LanceDB's `create_index` auto-names a single-column index `<column>_idx` regardless of index type — so building FM-Index on `text` after FTS already occupies `text` silently **replaced** the FTS index (both defaulted to `text_idx`) instead of erroring. `list_indices()` showed only one entry after both calls ran; the existing FTS-only search path broke (`Cannot perform full text search unless an INVERTED index has been created`) until this was caught by the coexistence test written for this spec.

**Fix:** pass an explicit distinct name to the FM-Index build — `.create_index(&["text"], Index::Fm(Default::default())).name("text_fm_idx".to_string())` — confirmed safe against the vendored crate source (`lancedb-0.31.0/src/index.rs:221`, `IndexBuilder::name`) and an upstream test proving multiple explicitly-named indices coexist (`lancedb-0.31.0/src/table/create_index.rs:892-925`). FTS keeps its default `text_idx`; only the second index needed disambiguating.

**Generalizable risk:** any future second index on a column that already carries a (possibly default-named) index needs an explicit distinct `.name(...)` or it silently clobbers the first, with no error at build time — only a downstream query failure. Worth remembering the next time this file adds an index to an already-indexed column.
