# ADR daemon-rework-003: Change gate uses git's algorithm, not git's state  [status: accepted]

- **Context:** After the event-kind hotfix (#265), the watcher still needs a content-change gate: fs events say "something happened", not "content changed". The user proposed using git rather than reinventing change detection. Grounding found git's oracle answers "differs from HEAD", but the watcher's question is "differs from what I last indexed" — those coincide only under commit-only indexing, which would change wiki freshness semantics (Obsidian/agent saves unsearchable until committed). Also several corpus roots are not git repos (~/.cheese, Maestral _meta).
- **Decision:** Adopt git's stat-cache *design* aimed at last-indexed state: on event, compare mtime against stored `FileSnapshot.mtime_ms` (skip free if equal); on mismatch, blake3 against stored `content_hash` (skip re-chunk if equal). `FileSnapshot{mtime_ms, content_hash}` already persists per file (plan.rs:9-18, lance.rs:1198-1239) — the gate hangs on existing storage. blake3 kept over git-blob-format SHA-1 (zero migration, faster, no interop need today).
- **Alternatives rejected:** commit-only indexing with git as oracle (freshness semantics change; still needs a non-git fallback); `git status` as pre-filter (dirty-but-already-indexed files re-trigger forever); epoch-based self-write suppression (unnecessary — reindex paths verified read-only w.r.t. corpus files: `rebuild_wiki_indexes` has exactly two callers, both explicit-write handlers).
- **Consequences:** uniform gate across git/non-git, tracked/untracked. Self-writes (INDEX-block updates) cost one stat compare, zero reads.
