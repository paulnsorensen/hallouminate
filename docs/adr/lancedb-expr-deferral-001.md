# ADR lancedb-expr-deferral-001: Defer escape_sql_str -> Expr migration  [status: accepted]

- **Context:** Issue #191 asked whether lancedb 0.31's DataFusion `Expr` predicate support (PR lancedb/lancedb#3444) could retire the hand-rolled `escape_sql_str` surface in `crates/hallouminate-adapters/src/lance.rs`. Research confirmed `Expr` support exists for query filters, `delete()`, and `merge_insert`, but `update()`/`UpdateBuilder` and `count_rows` remain SQL-string-only in 0.31.0 (`table/update.rs:26-45`, `table.rs:949`), so full retirement isn't possible today. A follow-up benchmark this session (`/tmp/expr-vs-sql-bench`, pinned `sqlparser 0.61.0`/`datafusion-expr 53.1.0` matching this workspace's `Cargo.lock`) measured the SQL-string parse path lance-datafusion actually runs (`lance-datafusion-6.0.0/src/sql.rs:44-46`) at ~4.7 μs/call versus ~0.14 μs/call for direct `Expr` construction — a real 35x ratio, but an absolute ~4.6 μs/call difference dwarfed by every affected call site's actual I/O/embedding cost (none of the six `escape_sql_str` call sites run in a per-row hot loop).
- **Decision:** Defer the Expr migration entirely. No predicate-building code changes to `lance.rs` in this pass. Issue #191 stays open, unlabeled and unchanged, for a future pass once upstream lancedb adds `Expr` support to `update()`/`count_rows`, or if a genuine hot-path use emerges.
- **Alternatives:** (a) Migrate the 5 Expr-capable call sites now, leaving `escape_sql_str` only for the 2 stuck sites — rejected: the measured performance case doesn't justify the migration effort/risk, and a partial migration leaves a permanent two-shaped predicate surface for no measured benefit. (b) Migrate now for injection-surface reduction alone (a non-performance rationale) — not evaluated in this pass; the triage discussion and this session's dialogue focused specifically on the performance question.
- **Consequences:** `escape_sql_str` and its six call sites are unchanged. The companion `fm-index-search-signal` spec even adds a seventh string-predicate call site (`contains(text, '<escaped query>')`), consistent with the existing pattern. Revisit trigger: upstream `update()`/`count_rows` Expr support ships, or a future profiling pass finds these call sites genuinely hot.
