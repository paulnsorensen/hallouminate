# ADR pressure-aware-maintenance-003: PSI probe with fail-open, activity-only portable fallback  [status: accepted]

- **Context:** Issue #230 requires deferring maintenance under elevated host I/O pressure and a defined portable fallback where Linux PSI is unavailable (macOS, pre-4.20 or non-CONFIG_PSI kernels, some containers). The codebase has no system-metrics dependency today.
- **Decision:** A crate-private `IoPressureProbe` trait with two impls: `PsiProbe` (Linux — parses `/proc/pressure/io` `some avg10`, elevated above 10%) and `NoPressureSignal` (never elevated). Absent or unparseable PSI is treated as not-elevated (fail-open) with a single debug log — a pressure-detection failure must never permanently block maintenance. Where PSI is unavailable, deferral rests on daemon-local activity alone. The trait seam doubles as the test-injection point required by the paused-time acceptance tests.
- **Alternatives:** (a) `sysinfo`-based heuristic (disk throughput / load average) — rejected: weak noisy proxy, new dependency. (b) Config-declared thresholds per host — rejected: pushes the measurement problem to the operator.
- **Consequences:** Zero new runtime dependencies; non-Linux hosts get activity-gating only, which is honest about what is measurable there. The 10% `some avg10` threshold is an uncalibrated starting value (spec `[TBD]`) — tune from production observations. Additional probe impls can be added behind the trait without touching the loop.
