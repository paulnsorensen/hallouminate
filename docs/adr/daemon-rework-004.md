# ADR daemon-rework-004: Crash-loop policy — persisted backoff, never give up  [status: accepted]

- **Context:** The internal watchdog's only reliable action on a frozen runtime is process abort, but a boot-deterministic wedge turns abort into a respawn storm (the 2026-07-17 watcher loop ignited in the first minute of every boot — such states are empirically real here). Research across systemd, k8s, OTP, launchd, Docker, MySQL/PostgreSQL/SQL Server found: no surveyed system auto-degrades after N crashes (degraded starts are manual operator opt-ins everywhere); the real split is backoff-forever (k8s CrashLoopBackOff, launchd ThrottleInterval) vs give-up-until-manual-clear (systemd failed state, OTP intensity exhaustion). launchd's default is the closest analog to our self-spawned, unsupervised shape; two live respawn-storm bug reports were both fixed by adding backoff.
- **Decision:** Trip ⇒ persist a timestamp to a trip-state file ⇒ clean abort. At boot, the daemon reads the file and honors an escalating backoff floor (10s doubling to a 5min cap) — within the floor it exits immediately with a distinct code. Never gives up permanently; never auto-degrades. Trip count and backoff state are exposed in StatusReport and the escalation ladder.
- **Alternatives rejected:** auto-degraded mode on Nth trip (zero precedent — mature systems keep retrying or make a human look, never silently self-reduce); give-up-until-manual-clear (grounding goes dark for every agent on the machine until a human notices); abort-only (respawn storm, the documented failure mode).
- **Consequences:** the self-persisted restart counter is precedented in principle but undocumented as a composition — we assemble, not invent. Boot backoff check brushes #220's spawn path (cold-start retry); kept separate. Loudness comes from telemetry, not denial of service.
